CVS Health — Information Security — Finding

Rule Construction Methodology, Intrusion Prevention Deployment, and SOC Readiness

Root Cause Analysis  •  PCI Cardholder Data Environment  •  Generated 2026-07-15
Finding: the PCI firewall estate's 48.0% IPS coverage rate — and the complete absence of coverage across 15 of 18 confirmed policies — is not an isolated gap to be closed by attaching more IPS policies to more rules. It is a direct, mechanical consequence of how the rules themselves are built. The ruleset is constructed as approximately 10,500 independent, ticket-scoped point rules rather than as a small number of policies governing categories of traffic. Intrusion prevention requires a category to reason about — a defined type of traffic to select an appropriate signature set for and make a blocking decision against. That category does not exist anywhere in this ruleset's construction. This is why coverage is missing where it's missing, and why coverage that does exist is confined to Detection mode rather than Prevention mode where it exists at all.

Why This Matters — In Plain Terms

An intrusion prevention system doesn't just look for "bad traffic" in the abstract. It looks for behavior that's abnormal for the type of transaction it's watching — and what counts as normal is completely different depending on who's talking to what.

A simple example: a large data transfer from a database server to an external IP address at 3 AM is either completely routine (a scheduled backup) or a sign of active data theft — and it's the exact same pattern on the wire either way. The only way to tell them apart is to know what kind of transaction this is supposed to be. A database server talking to a known backup destination on a known schedule is Machine→Machine traffic behaving normally. The same server suddenly pushing data somewhere it's never gone before is the same category of transaction behaving abnormally. An inspection engine that isn't told the difference between "this is Machine→Machine backup traffic" and "this is something else" can't make that call — it either has to let everything through unexamined, or flag so much routine activity that the real alerts get lost in the noise.

The same logic applies to every category in this finding: SSH from an admin's known workstation to a server is expected and low-risk — the same SSH connection initiated by the server itself, going somewhere new, is one of the clearest signs of a system already being used by an attacker to move further into the network. A user's browser downloading a large file from an approved SaaS application is normal; an internal application server doing the same thing to an unfamiliar destination is not. Same protocol, same basic pattern, completely different meaning — and the only way an inspection system can tell them apart is if the traffic is already sorted into "what kind of transaction is this" before the inspection decision gets made. That sorting is exactly what's missing here. Every rule in this estate treats a connection as just "this IP to that IP on this port" — with no record of whether it's a user reaching an app, one system reaching another, or a person reaching into infrastructure directly. Without that distinction, there's no way to write an inspection policy that's strict where it should be strict and permissive where it should be permissive — which is the specific, mechanical reason the choice observed in the data keeps coming out as "don't inspect it" or "watch it but don't touch it," rather than genuine, confident blocking.

The Causal Chain

1. Rule Origin Confirmed

Each rule is created to satisfy one access request — a ticket, a RITM, a Tufin change number — scoped to exactly the source, destination, and port that request specified. Nothing broader, nothing categorized.

↓

2. No Traffic-Type Grouping Exists Confirmed

Because each rule is a standalone point-to-point grant, there is no construct in the ruleset — no App-ID, no SGT, no address group functioning as a real category — that groups rules by what kind of traffic they represent (user access, service-to-service, infrastructure, admin).

↓

3. IPS Policy Assignment Has No Category to Attach To Confirmed mechanism

Both Cisco Secure Firewall and Palo Alto attach intrusion prevention at the rule or policy level, against a defined set of traffic — the signature set and blocking posture appropriate for "user web traffic" is different from what's appropriate for "database replication" or "SNMP polling." Choosing correctly requires knowing which is which. With ~10,500 undifferentiated point rules, that decision would have to be made individually, one rule at a time, for a ruleset where 68.5% of rules don't even self-describe what they're for.

↓

4. Two Outcomes, Both Observed in the Data Confirmed outcome

Outcome A — IPS is simply not attached. 15 of 18 Tier-1 policies, including the entire Middletown/Windsor tier, show 0% coverage. Outcome B — a separate, untuned catch-all is built instead. The five "(IPS Only)" policies found (Hawaii, Las Vegas Voyager, Nexus Secure PCI, Pittsburgh Epsilon, San Antonio) each apply one blanket "Inspect All" rule with no source/destination scoping at all — the closest thing to a workaround for the missing categorization.

↓

5. Why the Catch-Alls Are Detection Mode, Not Prevention Inferred — most consistent explanation, not directly confirmed

All five "(IPS Only)" policies run in Detection mode rather than Prevention. Detection alerts; it doesn't block. The most consistent explanation for that choice, given everything else found here: nobody can be confident what "normal" looks like for a blanket catch-all applied across thousands of uncharacterized point-to-point flows, so nobody is willing to risk blocking a ticket-approved connection that's never been individually characterized against a signature set. Prevention mode without traffic-type awareness carries real business-disruption risk; Detection mode is the safe default when the underlying ruleset can't tell you what should be blocked. This step is presented as the most plausible reading of the pattern, not as a confirmed statement from whoever configured these policies — worth a direct conversation with that team to confirm or correct.

The Next Problem This Creates — the SOC Has No Playbook for Alerts That Don't Exist Yet

Fixing the architecture doesn't automatically fix security by itself — it creates a new, immediate requirement that has to be met in parallel, not afterward. Today, with 15 of 18 policies at 0% IPS coverage and the rest either Detection-only or narrowly applied, the SOC has never had to systematically investigate and respond to intrusion alerts at the volume or variety this environment should actually be producing. There is effectively no playbook for what doesn't currently exist to alert on. That's not a criticism of the SOC — nobody can build a response process for an alert type they've never received.

The risk is what happens the moment that changes. As IPS coverage gets built out through the target-state model — Machine→Machine first, then User→Machine, then the two application quadrants — each of those phases will start generating alert types and volumes the SOC has no prior experience with. Without a playbook ready before that traffic goes live under Prevention mode, the realistic outcome isn't better security — it's alert fatigue, inconsistent triage, and real incidents getting lost in noise the SOC was never prepared to sort through. A technical fix that outpaces the team's ability to operate it doesn't improve the security posture; it just moves where the failure happens, from "no alert fired" to "an alert fired and nobody knew what to do with it."

The playbooks need to be built along the same four categories as everything else in this finding, because the correct response is different for each one:

QuadrantWhat an Alert Likely MeansResponse Pattern
Machine → MachineAn infrastructure system doing something outside its known backup/monitoring/sync patternCheck against a known-schedule baseline first; false positives are common here if the baseline isn't defined, but a genuine hit often means an already-compromised system being used to move data or reach further in
User → MachineAdministrative access happening outside expected patterns, or from an unexpected sourceHighest default severity — this is privileged access by definition. Immediate escalation, not a queue item, given how narrowly this quadrant is supposed to be scoped in the first place
User → AppA user account behaving in a way that doesn't match normal application usageVerify against the user's actual role and recent activity; may involve confirming with the person's manager or resetting credentials — this is the quadrant most likely to involve an actual person, not just a system
App → AppA service account or application tier reaching somewhere it doesn't normally goVerify with the application owner whether this is a legitimate new integration or a sign of lateral movement between tiers — the tier-to-tier design in the target state makes this easier to baseline than it is today

This is also why the discovery step already defined in the Target-State Policy Design Standard's build process matters for more than just writing the new rule. The same traffic-pattern data gathered to build a Machine→Machine rule correctly is the same data needed to write that quadrant's SOC playbook — what normal volume and timing looks like, so the playbook can define a real threshold instead of a guess. Playbook development should be sequenced as a parallel track to each phase of the IPS rollout, using that same discovery data, finished before — not after — that quadrant goes live under Prevention mode. Building the detection capability and building the team's ability to act on it are the same project, not two separate ones.

Why This Reframes the Remediation

The practical consequence of this chain: attaching more IPS policies to more existing rules does not fix this. It would mean making 10,500 individual, under-informed categorization decisions — the same operational bottleneck that produced 0% coverage in the first place, just repeated in the other direction. Deploying IPS meaningfully requires the traffic to already be grouped into reasoned-about categories before an inspection decision is attached to it — which is exactly what the Target-State Policy Design Standard's four-quadrant object model (User-Subnet-Group, App-Tier-Group, Infrastructure-Group, Admin-Subnet-Group) is for. Once traffic is grouped by what it actually is, "attach Prevention-mode IPS to all User→App traffic" becomes a single policy statement instead of an intractable per-rule audit. This finding is the direct justification for why that rebuild is necessary rather than optional — it isn't an architectural preference, it's the specific missing precondition for intrusion prevention to be deployable at all.
